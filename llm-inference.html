LLM Inference & Serving Optimization

The Inference Lifecycle

What happens from the moment you hit "send" to when you see the first token.

Every LLM API call goes through two distinct phases. Understanding these phases is essential because they have completely different performance characteristics and bottlenecks.

Two Phases of Inference

Phase 1: Prefill

Process the entire input prompt in parallel. Compute attention across all input tokens simultaneously.

Compute-bound — GPU math units are the bottleneck
All input tokens processed in parallel (matrix multiply)
Builds the KV cache for all input tokens
Determines Time to First Token (TTFT)
Phase 2: Decode

Generate output tokens one at a time. Each new token requires attending to ALL previous tokens.

Memory-bound — GPU memory bandwidth is the bottleneck
Tokens generated sequentially (autoregressive)
Reads the entire KV cache at every step
Determines Inter-Token Latency (ITL)

Key Latency Metrics

Responsiveness

TTFT

Time to First Token. How long the user waits before seeing any output. Dominated by prefill time. Target: <500ms for interactive apps.

Smoothness

ITL

Inter-Token Latency. Time between consecutive output tokens. Determines streaming smoothness. Target: <50ms for a "typing" feel.

Capacity

Throughput

Total tokens/second across all concurrent requests. The metric that determines cost-per-token and how many users you can serve.

The KV Cache

The single most important concept for understanding inference optimization.

During generation, each new token needs to "attend to" (look at) every previous token. Without caching, the model would recompute the Key and Value matrices for ALL previous tokens at every single generation step. The KV cache stores these computed K and V tensors so they only need to be computed once.

Why the KV Cache Exists

# Without KV cache (wasteful): recompute everything at each step for step in range(output_len): # Recompute K, V for ALL tokens (input + previously generated) K = compute_keys(all_tokens) # O(n) work repeated every step V = compute_values(all_tokens) # growing cost as sequence grows attn = softmax(Q @ K.T) @ V # With KV cache (efficient): only compute K, V for the new token for step in range(output_len): K_new = compute_keys(new_token) # O(1) work per step V_new = compute_values(new_token) kv_cache.append(K_new, V_new) # store for future steps attn = softmax(Q @ kv_cache.K.T) @ kv_cache.V

MHA vs MQA vs GQA

The biggest lever for reducing KV cache size is reducing the number of KV heads. Multi-Head Attention (MHA) uses one KV head per attention head. Grouped-Query Attention (GQA) shares KV heads across groups. Multi-Query Attention (MQA) uses a single KV head for all query heads.

Batching Strategies

How to process multiple requests simultaneously without wasting GPU cycles.

A single decode step uses a tiny fraction of the GPU's compute capacity — it's memory-bound, not compute-bound. Batching multiple requests together amortizes the cost of reading model weights from GPU memory, dramatically increasing throughput.

Static vs Continuous Batching

GPU stays busy processing actual tokens
Iteration-level scheduling: decide batch composition per step
Typical GPU utilization: 80-95%

PagedAttention & vLLM

Treating KV cache like virtual memory — the breakthrough behind vLLM.

Traditional inference engines pre-allocate KV cache as contiguous memory blocks for the maximum sequence length. This wastes massive amounts of GPU memory — a request that generates 100 tokens still reserves memory for 4096 tokens. PagedAttention solves this by allocating KV cache in small fixed-size pages, just like how operating systems manage virtual memory.

How PagedAttention Works

1. Page Table: Each request has a logical-to-physical page mapping, just like a CPU's virtual memory page table. Logical KV cache is contiguous from the model's perspective, but physical blocks can be anywhere in GPU memory.

2. On-Demand Allocation: New pages are allocated only when needed (as the sequence grows), not pre-allocated for the max length.

3. Memory Sharing: For parallel sampling (beam search, best-of-n), multiple sequences can share the same physical pages for their common prefix. Only divergent suffixes need separate pages (copy-on-write).

Speculative Decoding

Use a small model to draft tokens, a large model to verify — 2-3x speedup for free.

The key insight: verifying whether k tokens are correct can be done in a single forward pass (parallel, like prefill), while generating k tokens requires k sequential forward passes. Speculative decoding exploits this asymmetry.

How It Works

1. Draft: A small, fast model (the "draft model") generates k candidate tokens quickly.

2. Verify: The large target model processes ALL k draft tokens in a single forward pass (parallel verification).

3. Accept/Reject: Accept matching tokens from left to right. At the first mismatch, reject that token and all subsequent ones. Resample the rejected position from the target model's distribution.

4. Guarantee: The output is mathematically identical to what the target model would produce alone. Zero quality loss.

Speedup Analysis

When It Helps Most

High acceptance rate (draft model agrees with target 70-90% of the time). Larger draft length k (5-8 tokens). When the target model is much larger (and slower) than the draft model.

When It Doesn't Help

Low acceptance rate (creative, high-temperature generation where draft diverges). Very small target model (draft overhead not worth it). Memory-constrained (need to fit both models in GPU memory).

Quantization for Serving

Shrink model weights and KV cache to serve bigger models on smaller GPUs.

Quantization reduces the precision of model weights from 16-bit floats to 8-bit or 4-bit integers. This reduces memory by 2-4x and speeds up inference by reducing memory bandwidth requirements. The quality loss is surprisingly small for well-calibrated quantization.

Weight Quantization Methods

Simple

INT8 (W8A8)

Quantize weights and activations to 8-bit. Near-lossless for most models. 2x memory reduction. Supported natively by most hardware (TensorRT, CUDA).

Popular

GPTQ (INT4)

One-shot weight quantization using a calibration dataset. 4x memory reduction. Good quality with 128 group size. The standard for self-hosted models.

Advanced

AWQ

Activation-aware weight quantization. Protects the 1% of weights that matter most (based on activation patterns). Often better quality than GPTQ at the same bit-width.

CPU/Edge

GGUF

llama.cpp format. Multiple quant levels (Q4_K_M, Q5_K_S, etc). Optimized for CPU inference. Enables running 70B models on a MacBook.

Flash Attention

Flash Attention isn't quantization, but it's the other critical memory optimization. It fuses the attention computation into a single GPU kernel, avoiding writing the full attention matrix to HBM (GPU memory). This reduces memory usage from O(n^2) to O(n) and speeds up attention 2-4x.

# Standard attention: materializes n x n attention matrix in HBM attn_weights = softmax(Q @ K.T / sqrt(d)) # O(n^2) memory output = attn_weights @ V # Flash Attention: tiled computation, never materializes full matrix # Processes in blocks, accumulates in SRAM (fast on-chip memory) output = flash_attention(Q, K, V) # O(n) memory # 2-4x faster, enables longer sequences on same hardware

Model Parallelism

Large models don't fit on a single GPU. Two main strategies for splitting them across GPUs:

Tensor Parallelism (TP)
Split each layer's matrices across GPUs
Each GPU computes part of every layer
Requires all-reduce after each layer (high bandwidth)
Best within a node (NVLink: 900 GB/s)
Linear memory reduction: TP=4 → 4x less per GPU
Pipeline Parallelism (PP)
Assign different layers to different GPUs
Each GPU runs a subset of layers sequentially
Only send activations between stages (lower bandwidth)
Works across nodes (InfiniBand: 400 Gb/s)
Introduces "pipeline bubbles" of idle time
