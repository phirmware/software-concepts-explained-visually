LLM Evaluation & Evals Pipeline

Why Evals Matter

The difference between a demo and a production AI system is evaluation.

Every AI engineer can build a prototype that looks impressive in a demo. The hard part — the part that defines staff-level work — is knowing whether your system actually works, and proving it systematically.

The Eval-Driven Development Loop

At top AI companies, the development cycle revolves around evals. You don't ship a prompt change without running it through your eval suite, just like you don't ship code without running tests.

Define the Task

Start with a clear specification of what your LLM system should do. What are the inputs? What does a good output look like? What are the edge cases and failure modes? Write this down before touching any code.

Example: "The system should extract structured event data from calendar invites. It must handle timezone conversions, recurring events, and multi-day events."

How Top Companies Think About Evals

Model Provider

Anthropic's Approach

Evals are first-class infrastructure. Separate eval teams, thousands of test cases per capability, automated regression gating on model releases.

Anthropic maintains eval suites for safety (RLHF alignment), capability (coding, math, reasoning), and character (helpfulness, honesty). Every model checkpoint is evaluated against these suites before any deployment decision. The eval suite is versioned and grows continuously — new failure modes discovered in production become new eval cases.

Applied AI

Production Teams

Feature-specific evals tied to business metrics. Every prompt change requires eval comparison showing no regressions.

Production teams at companies like Stripe, Notion, and Cursor maintain task-specific eval suites. A typical setup: 200-500 golden examples per feature, LLM-as-judge scoring on a rubric, automated comparison in CI. No prompt change ships without a green eval run showing improvement or no regression on the metrics that matter.

Infra

Eval Infrastructure

Dedicated eval platforms — Braintrust, Humanloop, LangSmith — because in-house eval tooling is a competitive advantage.

The eval platform is treated like the testing framework: it handles dataset management, parallel execution, scoring aggregation, regression detection, and reporting. Top teams build custom eval harnesses that integrate with their CI/CD pipeline, often running evals on every PR that touches prompt templates or model configuration.

Types of Evaluations

Each eval type trades off cost, speed, and correlation with real quality.

There is no single "best" eval. Production systems use a layered approach — fast automated metrics for CI, LLM-as-judge for nuanced assessment, and targeted human evaluation for calibration.

Automated Metrics

Reference-based metrics that compare model output against a known-correct answer. Fast, cheap, deterministic — but limited to tasks where you have gold-standard references.

When to use automated metrics

Good for: Translation (BLEU), summarization (ROUGE), QA extraction (F1/EM), classification (accuracy/F1)

Bad for: Open-ended generation, creative tasks, anything where multiple valid answers exist. A paraphrase that conveys the same meaning can score 0 on exact match and poorly on BLEU.

LLM-as-Judge

Using a capable model (like Claude or GPT-4) to evaluate the output of another model. This is the most important eval technique for production AI systems — it scales like automated metrics but captures nuance like human evaluation.

The Judge Prompt Pattern

# Typical LLM-as-Judge prompt structure SYSTEM: You are an expert evaluator. Score the following response on a scale of 1-5 for each criterion. RUBRIC: Accuracy (1-5): Does the response contain factual errors? 1 = Major factual errors 3 = Minor inaccuracies 5 = Completely accurate Completeness (1-5): Does it address all parts of the query? 1 = Misses key aspects 3 = Covers main points 5 = Comprehensive coverage Relevance (1-5): Is the response focused on the question? 1 = Off-topic 3 = Partially relevant 5 = Directly addresses the query USER QUERY: {query} MODEL RESPONSE: {response} REFERENCE (optional): {reference_answer} Provide your evaluation as JSON: {"accuracy": N, "completeness": N, "relevance": N, "reasoning": "..."}

Key Design Decisions for LLM-as-Judge

Pointwise vs Pairwise

Pointwise: Score each response independently on a rubric (1-5). More stable but harder to calibrate.

Pairwise: Show two responses side by side, ask which is better. More reliable for ranking but O(n^2) comparisons.

Calibration & Bias

Position bias: Judges favor the first response in pairwise evals. Mitigate by swapping order and averaging.

Verbosity bias: Longer responses scored higher. Add "brevity is acceptable" to rubrics.

Human Evaluation

The gold standard — but expensive and slow. Use sparingly for calibrating your automated evals and for tasks where human judgment is the only valid signal.

Human Eval Patterns

Common

Likert Scale Rating

Annotators rate responses on a 1-5 scale for specific dimensions (quality, helpfulness, safety). Simple but requires clear rubric calibration.

Preferred

A/B Preference

Show two responses, ask which is better. More reliable than absolute scoring. Used by Anthropic and OpenAI for RLHF data collection.

Advanced

Inter-Annotator Agreement

Measure Cohen's kappa or Krippendorff's alpha across annotators. If annotators disagree, your rubric is ambiguous — fix it before trusting scores.

Code Evaluations

For code generation tasks, execution-based evaluation is the gold standard. Don't compare strings — run the code and check if it works.

Pass@k Metric

Generate k code samples, check how many pass the test suite. Pass@1 measures single-attempt success. Pass@10 measures the chance that at least one of 10 attempts works.

# Pass@k calculation # n = total samples generated # c = number that pass # k = number of attempts allowed def pass_at_k(n, c, k): """Unbiased estimator of pass@k""" if n - c < k: return 1.0 return 1.0 - comb(n - c, k) / comb(n, k) # Example: generated 20 samples, 8 pass tests pass_at_k(20, 8, 1) # = 0.40 (40% single-attempt success) pass_at_k(20, 8, 10) # = 0.99 (99% with 10 attempts)

Execution-Based Eval Pipeline

Sandboxed Execution

Run generated code in isolated containers (Docker, gVisor, Firecracker). Prevent filesystem access, network calls, and resource exhaustion. Time-limit execution (typically 10-30s).

Test Harness Design

Write test cases that check behavior, not implementation. Use property-based tests when possible. Include edge cases: empty inputs, large inputs, Unicode, error handling.

Safety Evaluations

Safety evals verify that your system handles adversarial inputs, refuses harmful requests, and doesn't leak sensitive information. This is non-negotiable for production systems.

Safety Eval Categories

Critical

Red-Teaming

Systematic adversarial testing — prompt injections, jailbreak attempts, social engineering. Automated red-team suites + human red-teamers for creative attacks.

Important

Toxicity & Bias

Measure output toxicity (Perspective API), stereotype reinforcement, demographic biases. Test across protected categories with paired prompts.

Compliance

Data Leakage

Verify the model doesn't memorize and regurgitate training data, PII, or secrets. Test with known canary strings and extraction attacks.

Advanced Patterns

Staff-level techniques that separate production-grade eval systems from toy setups.

Eval-Driven Development (EDD)

Like Test-Driven Development, but for AI. Write evals BEFORE changing your prompts or model configuration. This ensures you have a baseline to compare against and know exactly what you're optimizing for.

Typical Workflow
1. Get feature request
2. Change prompt
3. Try a few examples
4. Ship it
5. Find out it broke something else
6. Scramble to fix regression
EDD Workflow
1. Get feature request
2. Write eval cases for the new behavior
3. Run evals — confirm new cases fail
4. Change prompt
5. Run evals — new cases pass, old don't regress
6. Ship with confidence

Bootstrapping Eval Datasets with LLMs

You need hundreds of eval cases but creating them manually is slow. Use an LLM to generate candidate eval cases, then human-review them for quality.

# Generate eval cases with an LLM BOOTSTRAP_PROMPT = """ Generate 20 diverse test cases for a calendar event extractor. Each test case should have: - input: A natural language message containing event information - expected_output: Structured JSON with the extracted event Include edge cases: - Ambiguous times ("next Tuesday") - Multiple events in one message - Timezone conversions - Recurring events - Cancellations and modifications Format as JSON array. """ # Then human-review to remove bad examples # and add cases the LLM didn't think of

Multi-Dimensional Scoring

Real systems need to balance multiple competing objectives. A response can be accurate but unhelpful, or helpful but verbose. Track dimensions independently.

A/B Testing Prompts in Production

When you can't decide between two prompt versions offline, run them both in production and measure real user outcomes.

Simple A/B Split

Route 50/50 traffic between two prompt versions. Measure user satisfaction (thumbs up/down), task completion rate, and downstream metrics. Need statistical significance (p < 0.05) before declaring a winner.

Multi-Armed Bandit

Thompson sampling: start equal, gradually shift traffic to the better-performing version. Minimizes regret — you expose fewer users to the worse version while still gathering enough data to be confident.
