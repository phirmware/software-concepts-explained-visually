RAG Architecture Deep Dive

The RAG Mental Model

Give the LLM access to knowledge it wasn't trained on — without retraining.

Retrieval-Augmented Generation solves three fundamental problems with LLMs: knowledge cutoff (the model doesn't know about events after training), hallucination (the model confidently generates incorrect facts), and domain specificity (the model doesn't have access to your private data).

The core idea is simple: before asking the LLM to generate a response, first retrieve relevant information from a knowledge base and include it in the prompt context.

Embeddings & Vector Space

How text becomes searchable numbers — the foundation of semantic retrieval.

An embedding model converts text into a dense vector (array of floating-point numbers) that captures semantic meaning. Similar texts produce vectors that are close together in this high-dimensional space.

How Embeddings Work

# Conceptually: text → numbers that capture meaning "How do I reset my password?" → [0.23, -0.41, 0.87, ..., 0.12] # 1536 dimensions "I forgot my login credentials" → [0.21, -0.38, 0.85, ..., 0.15] # close to above! "What is the weather today?" → [-0.65, 0.72, -0.11, ..., 0.44] # far from both

Cosine Similarity

The standard distance metric for embeddings. It measures the angle between two vectors, ignoring magnitude. A cosine similarity of 1.0 means identical direction (semantically identical), 0 means orthogonal (unrelated).

# Cosine similarity calculation import numpy as np def cosine_similarity(a, b): return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b)) # "reset password" vs "forgot credentials" → 0.92 (very similar) # "reset password" vs "weather today" → 0.13 (unrelated)

Chunking Strategies

How you split documents determines retrieval quality more than anything else.

Chunking is the most underestimated part of RAG. The same retrieval system can go from 60% to 90% accuracy just by changing the chunking strategy. The goal: each chunk should be a self-contained unit of meaning that can answer a question on its own.

Chunking Trade-offs

Too Small
Chunks lose context — "it" refers to something in the previous chunk
Need more chunks to answer a question, eating context window
Embedding quality degrades for very short text
More chunks = more storage, slower retrieval
Too Large
Relevant info diluted with irrelevant content
Embedding represents a blend of topics — less precise matching
Fewer chunks needed but each wastes more context space
Harder to attribute answers to specific source passages

Production Recommendations

Start with: Recursive character splitting at 500-1000 tokens with 50-100 token overlap.

For structured docs: Use document-aware chunking that respects headings, code blocks, and tables.

For accuracy-critical: Semantic chunking with embedding similarity thresholds.

Always: Add overlap between chunks so context isn't lost at boundaries.

Vector Stores & Indexing

How vector databases make similarity search fast at scale.

Brute-force cosine similarity over millions of vectors is too slow. Vector stores use approximate nearest neighbor (ANN) algorithms to trade a small amount of accuracy for massive speed gains — typically 99% recall in milliseconds instead of minutes.

HNSW: The Dominant Algorithm

Hierarchical Navigable Small World graphs are the most widely-used ANN algorithm. Think of it as building a network of express highways on top of local roads.

Hybrid Search: Vector + Keyword

Vector search finds semantically similar content but can miss exact keyword matches. BM25 (keyword search) finds exact terms but misses paraphrases. Hybrid search combines both for the best of both worlds.

Re-Ranking & Post-Retrieval

Initial retrieval is fast but imprecise. Re-ranking makes it accurate.

Bi-encoder retrieval (embedding similarity) processes query and document independently — it's fast but can't capture fine-grained interactions between them. Cross-encoder re-ranking processes query and document together, catching nuances that bi-encoders miss.

Bi-Encoder vs Cross-Encoder

Bi-Encoder (Retrieval)
Encode query and docs independently
Compare with cosine similarity
Fast: O(1) per comparison (pre-computed)
Scales to millions of documents
Misses token-level interactions
Cross-Encoder (Re-Ranking)
Process [query, doc] pair together through the model
Outputs a single relevance score
Slow: O(n) forward passes, one per doc
Only practical for top-k results (k=20-100)
Captures token-level cross-attention

The Lost-in-the-Middle Problem

LLMs pay more attention to information at the beginning and end of the context window. Relevant information placed in the middle is often ignored. After re-ranking, place the most relevant chunks first and last in the prompt.

Common Failure Modes

RAG systems fail in predictable ways. Knowing the failure modes is half the battle.

Production RAG debugging is 80% of the work. When the system gives a wrong answer, you need to quickly identify whether the failure is in retrieval, context assembly, or generation.

The RAG Triad: Three Points of Failure

Context Relevance

Are the retrieved chunks actually relevant to the query? Measure: % of retrieved chunks that are relevant.

Groundedness

Is the answer actually supported by the retrieved context? Or is the LLM hallucinating beyond what the documents say?

Answer Relevance

Does the answer actually address the user's question? Even with good retrieval, the model might go off-topic.

Retrieval Failure

Query-Document Mismatch

User's phrasing doesn't match the document's vocabulary. Query: "How do I cancel my plan?" but docs say "subscription termination."

Root Cause: Embedding models capture semantics but can miss domain-specific synonyms. The gap between user language and document language is too large.

Fix: Use query expansion (generate alternative phrasings), hybrid search (BM25 catches keyword matches), or fine-tune your embedding model on domain-specific pairs. Add a query rewriting step that translates user language into document language.

Retrieval Failure

Wrong Granularity

Chunks are too large (relevant info diluted) or too small (context lost). The right answer spans two chunks but neither is retrieved.

Root Cause: Chunk boundaries don't align with semantic boundaries. Important context is split across chunks, and the overlapping window isn't large enough to capture it.

Fix: Use parent-document retrieval (retrieve the small chunk, return the larger parent). Increase chunk overlap. Use semantic chunking that splits at topic boundaries. Index at multiple granularities.

Context Failure

Context Window Overflow

Too many chunks stuffed into the prompt, exceeding the context window or pushing out the system prompt and instructions.

Root Cause: Retrieving too many chunks (top-20 instead of top-5), or chunks are too large. The total context exceeds the model's effective attention span even if it fits the token limit.

Fix: Re-rank and take only top-3 to top-5 most relevant chunks. Use contextual compression to extract only the relevant sentences from each chunk. Set a hard token budget for retrieved context (e.g., max 2000 tokens).

Context Failure

Lost in the Middle

Relevant information is in the retrieved context but the model ignores it because it's buried in the middle of a long prompt.

Root Cause: LLMs have a U-shaped attention pattern — they attend most to the beginning and end of the context. Information in the middle gets less attention.

Fix: Place the most relevant chunk first. Use the "sandwich" pattern: most relevant at start, second-most relevant at end, less relevant in the middle. Alternatively, reduce total context length so the model can attend to everything.

Generation Failure

Hallucination Despite Context

The model generates facts not present in the retrieved context, or contradicts the context with its parametric knowledge.

Root Cause: The model's parametric knowledge (from training) conflicts with the retrieved context, and the model trusts its own knowledge more. Or the model extrapolates beyond what the context says.

Fix: Add explicit instructions: "Only answer based on the provided context. If the context doesn't contain the answer, say so." Use a grounding check as a post-processing step (LLM-as-judge verifying the answer is supported by context). Lower temperature for more conservative generation.

Data Failure

Stale or Conflicting Data

The vector store contains outdated documents, or multiple documents contradict each other on the same topic.

Root Cause: No document lifecycle management. Old versions of docs are still indexed alongside new versions. Or multiple sources of truth exist for the same information.

Fix: Implement document versioning and deduplication in your ingestion pipeline. Add metadata timestamps and filter for recency. Use a reconciliation step when multiple chunks contradict each other. Build an incremental update pipeline rather than full re-indexing.
